методология проектирования программных систем, которая представляет программу в виде совокупности объектов, которые являются экземплярами классов. Эти объекты взаимодействуют - отправляя друг другу сообщения (вызывая методы)ю ООП фокусируется на моделировании предметной области через ее ключевые сущности, их свойства и поведение.
Это основной подход к проектированию, пришедший на смену структурному и тесно связанный с языками программирования (C++, Java, C#, Python и другие). Это ключевой инструмент на этапах детального проектирования и конструирования системы.

---
### **Фундаментальные концепции («Три кита» ООП)**
1. **Инкапсуляция** - данные (поля, атрибуты) и методы (функции) объединяются в единую структуру (классы). Это позволяет скрыть внутреннюю реализацию от внешнего мира. Позволяет управлять доступом, повышать надежность, обеспечить целостность данных. Является продолжением идеи черного ящика из структурного программирования. Снижает связанность
2. **Наследование** - механизм для создания нового класса (потомка, дочернего класса) на основе существующего класса (родительского, базового класса). Потомок наследует структуру и поведение родителя. Это позволяет повторно использовать код, создавать иерархии классов.
3. **Полиморфизм** - позволяет объектам с одним интерфейсом иметь разную реализацию внутри. К примеру, можно использовать объект дочернего класса вместо родительского, или делать перегрузку методов. Это повышает гибкость и расширяемость. Позволяет проектировать общие алгоритмы

---
### **Ключевые элементы ООП-проектирования**

1. **Класс** - шаблон или описание для создания объекта. Определяет атрибуты (данные) и методы (поведение), которые будут у объекта. Это основная единица проектирования. Хорошо - высокая связность
2. **Объект или Экземпляр** - конкретная реализация класса, созданная в памяти во время выполнения программы. Имеет конкретное состояние
3. **Абстрактный класс и Интерфейс**:
	**Абстрактный класс** - класс, который нельзя инстанировать, он только для наследования. Может содержать абстрактные методы
	**Интерфейс** - Полностью абстрактная структура, определяющая только контракт, который потом должны реализовать классы. Обеспечивает слабую связанность и полиморфизм.
	Нужны для определения общих черт для семейства классов, проектирование на уровне абстракций.
4. **Отношения между классами**:
	**Ассоциация** - Факт связи между классами (например, `Преподаватель` читает `Дисциплину`)
	**Агрегация** - отношение часть-целое, где часть может существовать независимо от целого (например, `Кафедра` содержит `Преподавателей`)
	**Композиция** - Более строгая форма агрегации, где часть не может существовать без целого (то есть, их жизненный цикл совпадает).  (Например, `Заказ` и позиции `Заказа`)
	**Зависимость** - Временная связь, когда один класс использует другой в методе (например, передача объекта в параметре)

### **Методы и инструменты ООП-проектирования (связь с курсом)**

1. **UML - основной язык моделирования**:
	**Диаграмма классов (Class Diagram):** Ключевой инструмент. Визуализирует классы, их атрибуты, методы и отношения между ними. Используется для статического проектирования архитектуры
	**Диаграмма последовательности (Sequence Diagram)** - моделирует динамическое взаимодействие объектов во времени
	**Диаграмма Use Case** - не совсем ООП, но позволяет выделить ключевые сущности и акторов.
2. **Принципы проектирования (SOLID, DRY, KISS)**
	S - **Single Responsibility** - Класс должен иметь только одну причину для изменения (высокая связность)
	O - **Open/Closed** - Классы должны быть открыты для расширения, но закрыты для модификации
	L -  **Liskov Substitution** - Объекты в программе должны быть заменяемыми на экземпляры их подтипов без изменения правильности программы.
	I -  **Interface Segregation** - Много специализированных интерфейсов лучше одного общего
	D - **Dependency Inversion** - Зависимости должны строиться на абстракциях, а не на деталях (низкая связанность) (к примеру, func distance(int x1, int y1, int x2, int y2) хуже, чем distance(Point a, Point b))
3. **Шаблоны проектирования (Design Patterns)** - повторяемые архитектурные решения типовых проблем в ООП (к примеру, Фабрика, Наблюдатель, Декоратор). Не надо изобретать велосипед

---
### **Связь с другими темами курса**

- **Структурное проектирование** - ООП наследует и углубляет его ключевые цели - управление сложностью через модульность. Класс - улучшенный модуль
- **Domain-Driven Design (DDD)**. ООП - основа для реализации DDD. 
- **Архитектурные шаблоны**: Многие современные архитектурные шаблоны (слоистая, шестиугольная, микросервисы) реализуются с использованием принципов ООП для организации кода внутри компонентов
- **Рефакторинг**: Большинство приемов рефакторинга (например, выделение метода, выделение класса, подъем поля) направлены на улучшение объектно-ориентированной структуры кода, повышение связности и снижение связанности.

**Вывод:**
Объектно-ориентированное проектирование — это не просто синтаксис языка, а **система мышления и комплексная методология** для создания сложных, адаптируемых и поддерживаемых систем. Оно предоставляет мощные средства для **инкапсуляции сложности**, **моделирования предметной области** и построения **гибких архитектур**. Владение принципами ООП, умение моделировать с помощью UML и применять шаблоны проектирования является обязательным навыком современного разработчика, напрямую влияющим на **качество** и **жизненный цикл** создаваемого программного обеспечения.
