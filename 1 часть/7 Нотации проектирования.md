**Нотация проектирования** — это система условных графических символов, правил их соединения и использования, предназначенная для создания формализованных моделей (диаграмм, схем) различных аспектов программной системы на этапах анализа и проектирования.

Главная цель нотаций — **визуализировать сложность**, обеспечить единое понимание системы всеми участниками проекта (аналитиками, разработчиками, заказчиками) и зафиксировать проектные решения в виде наглядных **артефактов**.

### **Классификация нотаций по целям и уровням моделирования (на основе лекций)**

#### **1. Нотации для моделирования предметной области и бизнес-процессов (Стадия анализа)**

Используются для понимания контекста, выявления требований и моделирования деятельности «как есть» (AS-IS) и «как должно быть» (TO-BE).

*   **IDEF0:**
    *   **Назначение:** Функциональное моделирование деятельности предприятия. Показывает основные функции (работы), их входы, выходы, управляющие воздействия и механизмы.
    *   **Особенности:** Иерархичность, строгая формальность. Хороша для описания управленческих процессов.
    *   **Контекст:** Используется в рамках **бизнес-архитектуры** и подхода **TOGAF** ([[Архитектура предприятия (TOGAF)]]).
![[IDEF0.png]]
*   **BPMN 2.0 (Business Process Model and Notation):**
    *   **Назначение:** Стандартная нотация для моделирования бизнес-процессов. Показывает поток работ (activities), события (events), шлюзы (gateways), участников (pools/lanes).
    *   **Особенности:** Интуитивно понятна, позволяет моделировать как простые, так и сложные процессы, вплоть до уровня исполняемых процессов.
    *   **Контекст:** Основной инструмент для **анализа бизнес-процессов** 
![[BPMN 2.0.png]]
*   **EPC (Event-Driven Process Chain):**
    *   **Назначение:** Моделирование бизнес-процессов с акцентом на события, которые инициируют и завершают функции.
    *   **Особенности:** Широко использовалась в методологии ARIS. Показывает связь «Событие – Функция – Событие».
![[EPC.png]]
*   **DFD (Data Flow Diagram) – Диаграмма потоков данных:**
    *   **Назначение:** Моделирование **информационных потоков** в системе. Показывает, как данные перемещаются между процессами, внешними сущностями и хранилищами данных.
    *   **Особенности:** Акцент на данные, а не на порядок операций. Помогает выявить «узкие места» в информационных потоках.
![[DFD.png]]
#### **2. Нотации для моделирования структуры и поведения системы (Стадии проектирования архитектуры и детального проектирования)**

*   **UML (Unified Modeling Language) – основной инструмент объектно-ориентированного проектирования:**
    UML включает множество типов диаграмм, которые делятся на **структурные** и **поведенческие**.

    **Структурные диаграммы UML (отвечают на вопрос «Из чего состоит?»):**
    *   **Диаграмма классов (Class Diagram):** Наиболее важная. Показывает классы, их атрибуты, методы и отношения между ними (ассоциация, наследование, агрегация, композиция). Основа для проектирования объектной модели.
    ![[Диаграмма классов.png]]
    *   **Диаграмма компонентов (Component Diagram):** Показывает компоненты системы (более физические, чем классы: модули, библиотеки, исполняемые файлы) и зависимости между ними. Внутренее устройство компонент спрятано. Соответствует **представлению уровня «Component» в подходе С4** ([[подход C4]]).
    * ![[Диаграмма компонент.png]]
    *   **Диаграмма развертывания (Deployment Diagram):** Показывает физическое развертывание артефактов на аппаратных узлах (серверах, рабочих станциях). Соответствует уровню «Container» в С4.
    ![[Диаграмма развертывания.png]]

    **Поведенческие диаграммы UML (отвечают на вопрос «Как работает?»):**
    *   **Диаграмма прецедентов или вариантов использования (Use Case Diagram):** Показывает взаимодействие акторов (пользователей, систем) с системой через варианты использования (Use Case). Используется на стыке анализа и проектирования.
    ![[Диаграмма Use Case.png]]
    *   **Диаграмма последовательностей (Sequence Diagram):** Детально моделирует взаимодействие объектов во времени, показывая порядок сообщений между ними. Критически важна для проектирования сложной логики.
    ![[Диаграмма последовательностей.png]]
    *   **Диаграмма состояний (State Machine Diagram):** Моделирует поведение объекта, показывая, как его состояние меняется в ответ на события.
    *   **Диаграмма активности (Activity Diagram):** Напоминает блок-схему, показывает поток операций (действий) и принятие решений. Может использоваться для моделирования бизнес-процессов или алгоритмов.

*   **ER-диаграммы (Entity-Relationship):**
    *   **Назначение:** Концептуальное и логическое моделирование данных. Показывает сущности (таблицы), их атрибуты (поля) и связи между ними (один-ко-многим и т.д.).
    *   **Контекст:** Прямой переход к проектированию базы данных (Лекция 4). Является частью **информационного обеспечения** системы.
![[ER диаграмма.png]]
#### **3. Нотации для проектирования архитектуры высокого уровня**

*   **Подход С4 (Context, Containers, Components, Code):** Фактически предлагает набор соглашений для создания диаграмм на разных уровнях абстракции
    *   **Диаграмма контекста (Context):** Система как «черный ящик» и её взаимодействие с пользователями и другими системами.
    *   **Диаграмма контейнеров (Containers):** Показывает высокоуровневые технологические блоки (веб-приложение, мобильное приложение, база данных, микросервисы) и их взаимодействие.
    *   **Диаграмма компонентов (Components):** Декомпозиция контейнеров на ключевые компоненты (логические модули).

*   **Диаграммы Константайна / Структурные диаграммы:**
    *   **Назначение:** Показывают **связность (cohesion)** и **связанность (coupling)** модулей. Помогают оценить качество структуры системы с точки зрения сопровождаемости.
    * Главная цель - выявление и анализ блокировок, задержек, взаимного ожидания и потенциальных узких мест в многозадачных или многопоточных системах.
![[Карта Константайна.png]]
*   **Диаграммы Джексона (JSD):**
    *   **Назначение:** Метод структурного проектирования, ориентированный на данные. Используется для проектирования структуры программы на основе структуры её входных и выходных данных (Лекция 4).
    * Главный принцип: «Структура программы должна отражать структуру её данных». Если вы правильно опишете, как выглядят входные и выходные данные, то сможете вывести из этого структуру самой программы.
    * Диаграмма Джексона моделирует иерархическую структуру данных с помощью древовидных диаграмм, а затем по этим же правилам строится и схема программы.
![[Диаграмма Джексона.png]]
*   **Свободная нотация для контекстных диаграмм:**
    *   **Назначение:** Быстрое и понятное изображение системы в её окружении, без строгих правил, но с ясностью для всех стейкхолдеров.

### **Критерии выбора нотации**

1.  **Цель моделирования:** Что мы хотим показать? (структуру, поведение, данные, процессы).
2.  **Аудитория:** Кто будет читать диаграмму? (бизнес-заказчик, архитектор, разработчик).
3.  **Уровень детализации:** Высокоуровневая архитектура или детальная реализация.
4.  **Принятые стандарты в команде/отрасли:** Например, BPMN для бизнес-аналитиков, UML для разработчиков.

### **Вывод**

Нотации проектирования — это не самоцель, а **инструменты коммуникации и анализа**. В современной практике проектирования используется **комбинация нотаций** для полного охвата всех аспектов системы. Например, для одного проекта могут использоваться:
*   **BPMN** — для анализа бизнес-процесса заказчика.
*   **Use Case Diagram** — для определения границ системы.
*   **Диаграмма контейнеров (C4)** — для выбора технологического стека.
*   **Sequence Diagram** — для проектирования сложного взаимодействия.
*   **Class Diagram и ERD** — для детального проектирования модулей и базы данных.

Грамотное владение нотациями позволяет перейти от текстовых, часто противоречивых, требований к однозначным, визуализированным моделям, которые становятся основой для успешной разработки и интеграции.
