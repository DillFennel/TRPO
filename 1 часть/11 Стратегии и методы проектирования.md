Стратегии и методы проектирования - это набор требований, подходов и конкретных приемов, которые нужны для преобразования исходных требований к ПО в детализированный проект, готовый к реализации.

### **I. Стратегии проектирования (общие подходы)**

Стратегии определяют философию и последовательность действий при проектировании.

1. **Нисходящее проектирование (Top-Down Design / Stepwise Refinement):**
	Начинается с общего представления системы (контекста, главной функции) и последовательно декомпозируется на более мелкие и простые подсистемы и модули, пока не будут достигнуты простые, легко реализуемые компоненты.
	Лежит в основе структурного проектирования и создания деревьев программных модулей.
	**Процесс:** Общая система → Подсистемы → Модули → Процедуры/функции
	**Преимущества:** Хорошо управляет сложностью, обеспечивает системный взгляд
	**Недостатки:** Риск неправильной декомпозиции на ранних этапах, что трудно исправить потом. Сильно зависит от первоначального понимания
2. **Восходящее проектирование (Bottom-Up Design):**
	Проектирование начинается с базовых, примитивных элементов (например, структур данных, утилитарных функций), которые затем комбинируются во всё более крупные и сложные подсистемы, пока не будет получена полная система.
	Часто применяется при интеграции готовых библиотек, сервисов или при рефакторинге, когда улучшают низкоуровневые модули
	**Процесс:** Элементарные компоненты → Составные модули → Подсистемы → Система.
	**Преимущества:** Хорошо для повторного использования существующих компонентов. Позволяет рано протестировать базовые элементы
	**Недостатки:** Сложно заранее увидеть общую архитектуру, может привести к плохой интеграции компонентов.
3. **Проектирование, управляемое данными (Data-Driven Design):**
	Структура программы и ее модулей выводится из структуры обрабатываемых данных. Ключевой вопрос: «Какие данные есть и какие преобразования с ними нужно сделать?».
	**Методы:** метод Джексона (JSD), построение ER-диаграмм и DFD
	**Применение**: Эффективно для систем, ориентированных на обработку информации (пакеты обработки, компиляторы, системы отчетности)
4. **Объектно-ориентированное проектирование (Object-Oriented Design):**
	Стратегия, при которой система моделируется как набор взаимодействующих объектов, объединяющих данные и поведение. Проектирование фокусируется на выявлении классов, их ответственности и взаимоотношений.
	Основная современная стратегия для детального проектирования
	**Методы:** Использование UML (диаграммы классов, диаграммы последовательностей), применение шаблонов проектирования и принципов [[SOLID]]
5. **Предметно-ориентированное проектирование (Domain-Driven Design - DDD):**
	Стратегия, при которой структура и язык программной модели (архитектура) напрямую отражают ключевые концепции и бизнес логику предметной области. Акцент делается на глубоком погружении в домен
	Это современный подход к преодолению концептуальной сложности, особенно при построении ядра (core) системы
	**Ключевые практики**: Построение [[Ubiquitous Language]] (единого языка), выделение ограниченных контекстов ([[Bounded Contexts]]), проектирование сущностей (Entities), объектов-значений (Value Objects), сервисов домена
### **II. Конкретные методы проектирования (инструменты)**

Это тактические приемы, используемые в рамках выбранных стратегий
1. **Метод структурного проектирования (Йордон-Константайн):**
	Позволяет создать модульную структуру с высокой связностью и низкой связанностью
	**Инструменты:** Анализ потоков данных (DFD), построение структурных схем (диаграмм Константайна) для визуализации связей между модулями
2. **Моделирование с использованием UML (Unified Modeling Language):**
	Позволяет визуализировать различные аспекты системы с помощью стандартизированных нотаций
	**Инструменты:** Набор диаграмм для статистического и динамического моделирования:
	- **Структурные:** диаграммы классов, компонентов, развертывания
	- **Поведенческие:** Диаграммы Use Case, последовательностей, состояний, активности
3. **Метод функциональной декомпозиции:**
	Позволяет разбить сложную функции системы на более простые, связанные между собой подфункции.
	**Инструменты:** Иерархические диаграммы функций, диаграммы IDEF0
4. **Прототипирование (эволюционное проектирование):**
	Быстрое создание упрощенной, но работоспособной версии продукта (или ее части). Позволяет проверить гипотезы, уточнить требования и проектные решения
	Ключевой элемент интегративных моделей ЖЦ и  подхода к созданию MVP.
5. **Архитектурные подходы (шаблоны высокого уровня):**
	Позволяют выбрать базовые структуру системы. Это стратегические методы для архитектурного проектирования.
	**Примеры**:
	- [[Слоистая архитектура]] (Layered Archettecture)
	- [[Микросервисная архитектура]]
	- Событийно-ориентированная архитектура ([[Event-Driven Architecture]])
	- [[Шестиугольная архитектура]] (Hexagonal / Ports and Adapters)
	- [[подход C4]] Model для представления архитектуры на разных уровнях
6. **Применение шаблонов проектирования (Design Patterns):**
	Решение типовых проблем проектирования с помощью проверенных документированных решений (например Фабрика, Наблюдатель, стратегия). Позволяют повысить гибкость и сопровождаемость кода.
### **Выбор стратегии и методов: факторы влияния**

Выбор зависит от:
- **Характера системы** - Обработка данных (data-driven) vs. сложная бизнес-логика (DDD, OOP).
- **Модели ЖЦ** - Каскадная (предполагает нисходящее, тщательное проектирование) vs. Agile/Итеративная (допускает прототипирование, восходящее наращивание через MVP).
- **Опыта команды** - Знание DDD, UML, шаблонов.
- **Требований** - Критичность нефункциональных требований (производительность, масштабируемость) диктует выбор архитектурного шаблона.

**Вывод:**
 Не существует единственно правильной стратегии. Успешное проектирование — это **комбинация подходов**: использование **DDD** для моделирования ядра бизнес-логики, **UML и OOP** для детального проектирования компонентов, **архитектурных шаблонов** для обеспечения требуемых качеств системы и **прототипирования** для валидации решений. Понимание всего спектра стратегий и методов позволяет архитектору и команде выбрать наиболее эффективный инструментарий для конкретного проекта.
